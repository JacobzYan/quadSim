Today was a relatively short day, There is a lot of work that needs to get done to just properly pull in quad parameters and I might as well do it properly right now so I have a good foundation for the future. These are all going in simulator/quadParams.cc/.h. My goal is to make this the class that is able to read in config files to set up all the parameters. I want to be able to pass in multiple files seperating out the constants that don't change, and the quad parameters that could be different between different quadcopters/could be corrected by some form of model correction.
- From that I need to be able to read in any number of files that contain params. (I'll have to do this with the argc and argv variables to read command line prompts)
- Since I'll be reading multiple files, I will probably prescribe that all relevant files live in the same folder as csvs.
	- Perhaps it would be convenient to have a default file (mainly thinking for the physical constants)
I originally started by just initializing every variable separately, but I realized that I wanted to have some level of customizability within the config files (throwing in a configuration with more props, a different sensor package), so I am now thinking of breaking those down into sub-classes. Since props and sensors both need to know their pose with respect to the quad body frame, I have created a template class to add those variables/related methods in so I don't need to copy them for every flavor of sensor I include. 

Coming up I still need to decide if I'm going to go with point rotation/alias or frame rotation/alibi methodology. I am more familiar with point rotation and it conceptually makes more sense to me (frame B's unit vectors are represented in frame A's coordinate), but the math is a little more annoying to do in that setup (needing to invert to rotate a vector by the matrix).
Come to think of it, I'm not even 100% sure that I'm using the correct assignments, this is something I definitely need to do more research on before I settle on something. 
I've also considered using quaternions for the core computation of the algorithm since there are fewer parameters to pass around, This will probably be a stretch goal after I get something working, I can duplicate my ode portion of the simulator using quaternions and look at the computational time for a long run. Too many stretch goals here, I should probably start tracking them somewhere... A problem for near future me perhaps, as I am tired and still have work tomorrow
